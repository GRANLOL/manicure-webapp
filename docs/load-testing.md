# Нагрузочное тестирование

Этот проект можно тестировать под нагрузкой по `FastAPI`-части, не пытаясь искусственно нагружать сам Telegram Bot API.

Что имеет смысл нагружать:

- `GET /api/health`
- `GET /api/get-content`
- `GET /api/busy-slots`
- `POST /api/bookings`

Самый полезный сценарий для этого проекта:

- массовое чтение слотов
- обычные конкурентные записи на разные слоты
- конфликтные записи в один и тот же слот

## Важное ограничение по авторизации

Если `WEBAPP_AUTH_REQUIRED=true`, API требует корректный `X-Telegram-Init-Data`.

Для нагрузочного теста есть 2 нормальных варианта:

1. Поднять отдельный тестовый стенд и временно поставить:
   - `WEBAPP_AUTH_REQUIRED=false`
2. Передать в `k6` реальный `TG_INIT_DATA`, снятый из тестового Telegram WebApp

Для первого реального прогона удобнее именно тестовый стенд с отключённой webapp-авторизацией.

## Рекомендованный безопасный стенд

Не стреляйте сразу по боевой базе.

Минимально безопасный вариант:

- поднять копию сервиса
- использовать отдельную тестовую `bookings.db`
- временно отключить клиентские уведомления или использовать тестового бота

Иначе вы:

- создадите мусорные записи
- получите реальные уведомления в Telegram
- загрязните отчёты и клиентскую историю

## Обязательные параметры запуска

Для сценариев записи задаются:

- `BASE_URL` — адрес API, например `https://api.tgbooking.online/api`
- `BOOKING_DATE` — дата записи в формате `дд.мм.гггг`
- `UNIQUE_TIMES` — список свободных слотов через запятую
- `CONFLICT_TIME` — один свободный слот для конфликтного теста
- `SERVICE_ID` или `SERVICE_NAME`

Опционально:

- `TG_INIT_DATA` — если авторизация WebApp включена
- `PHONE_PREFIX` — префикс для тестовых телефонов

## Пример: тест чтения + записи

```powershell
k6 run `
  -e BASE_URL=https://api.tgbooking.online/api `
  -e BOOKING_DATE=05.04.2026 `
  -e SERVICE_ID=153 `
  -e UNIQUE_TIMES=10:00,10:30,11:00,11:30 `
  -e CONFLICT_TIME=12:00 `
  scripts/load_test_k6.js
```

Если нужен `X-Telegram-Init-Data`:

```powershell
k6 run `
  -e BASE_URL=https://api.tgbooking.online/api `
  -e TG_INIT_DATA="query_id=...&user=...&auth_date=...&hash=..." `
  -e BOOKING_DATE=05.04.2026 `
  -e SERVICE_ID=153 `
  -e UNIQUE_TIMES=10:00,10:30,11:00,11:30 `
  -e CONFLICT_TIME=12:00 `
  scripts/load_test_k6.js
```

## Что делает сценарий

### `browse`

Имитация обычного открытия приложения:

- `health`
- `get-content`
- `busy-slots`

По умолчанию:

- `10` виртуальных пользователей
- `30s` длительность

Меняется через:

- `BROWSE_VUS`
- `BROWSE_DURATION`
- `BROWSE_SLEEP`

### `unique_booking`

Параллельные записи на разные слоты.

По умолчанию:

- число `VU` берётся из количества `UNIQUE_TIMES`
- каждая VU делает одну попытку

Меняется через:

- `UNIQUE_BOOKING_VUS`
- `UNIQUE_BOOKING_ITERATIONS`

### `conflict_booking`

Параллельные записи в один и тот же слот.

Ожидаемое поведение:

- один запрос может пройти успешно
- остальные должны получить корректный отказ `409`
- не должно быть дублей в базе

Меняется через:

- `CONFLICT_VUS`
- `CONFLICT_ITERATIONS`

## Как интерпретировать результат

Смотреть в первую очередь на:

- `http_req_failed`
- `http_req_duration`
- `p95`
- ответы `409` в конфликтном сценарии

Для этого проекта хороший знак:

- чтение работает стабильно
- конфликтные записи дают предсказуемые `200/409`
- нет хаотичных `500`
- после теста в базе нет дублей на один слот

## Что проверить после прогона вручную

1. Сколько реально создалось тестовых записей.
2. Не появилось ли дублей на один и тот же слот.
3. Нет ли ошибок `database is locked` в логах.
4. Не ушли ли лишние уведомления клиентам.

На VPS полезно смотреть:

```bash
journalctl -u bookingbot -n 200 --no-pager
```

И параллельно:

```bash
top
```

или:

```bash
htop
```

## Рекомендованная лестница нагрузки

Не начинайте сразу с больших чисел.

Нормальный порядок:

1. `10` VU
2. `30` VU
3. `50` VU
4. `100` VU

И отдельно:

- `10-20` одновременных конфликтных запросов на один слот

Для текущего масштаба проекта этого уже достаточно, чтобы понять реальную устойчивость.
